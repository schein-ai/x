How we ship AI agents that hold up in production.

A framework for prompt scaffolding, tool-call contracts, evaluation harnesses, and operational rituals — drawn from four production engagements between 2025 Q3 and 2026 Q1.

Abstract

Many AI agent demos work in the demo and break in production. The reasons are familiar: prompts that drift under unexpected input, evaluation runs that pass for the wrong reasons, tool integrations that look connected but are silently unobserved. This write-up documents the framework we use to ship AI agents that survive contact with real business operations.

We describe four components — (i) modular prompt scaffolding, (ii) typed tool-call contracts with negative-example coverage, (iii) an evaluation harness with explicit failure-mode buckets, and (iv) operational rituals (logging, replay, human-in-loop overrides) for the deployed agent. The framework was applied across four production engagements in 2025 Q3 – 2026 Q1, including a knowledge-search agent over twenty years of JP-language internal documentation. We report the metrics we track, the failure modes we observed, and the operational changes we made between v0.1 and v0.4 of the harness. The harness itself is released under MIT licence as agent-eval.

§ 1 — IntroductionThe gap between demo and deployment.

Generative AI agents — language-model-driven systems that plan, call tools, and produce structured outputs — have shipped in demos faster than in production. The gap between the two settings is not primarily a model-capability gap. We argue, on the basis of four production engagements between 2025 Q3 and 2026 Q1, that the gap is operational: agents fail in production because the framework around the agent is missing, not because the underlying model is insufficient.

The literature on agent design [1][2] emphasises planning architectures and tool-use protocols. Less attention has been paid to the operational substrate: how prompts are versioned, how tool calls are contractually constrained, how evaluation handles negative cases, and how the deployed agent is observed at runtime. Each of these is a place where production agents fail silently. Each is, in our experience, fixable with disciplined engineering rather than further model capability.

This write-up describes the framework we have converged on. It is deliberately not novel: the components are recognisable to any infrastructure-minded engineer. The contribution is in their combination and operational tuning, validated across four real engagements with paying clients. The framework's evaluation harness is open-sourced as agent-eval under MIT licence.

§ 2 — BackgroundThree failure modes we keep seeing.

Before describing the framework, it is worth being concrete about what fails. Across the four engagements, three failure modes accounted for the majority of observed production incidents:

  1. Prompt drift under input variance. A prompt that performs well on the curated eval set degrades on real user input distributions. The model produces plausible but incorrect outputs that pass automated checks because the checks were calibrated to the eval set, not production traffic.
  2. Silent tool-call mismatch. The agent emits a structured tool call that the orchestrator parses successfully but the downstream tool rejects (or, worse, accepts and produces wrong output). Because the orchestrator-side parse succeeds, the agent assumes success and proceeds.
  3. Evaluation passing for the wrong reasons. The eval suite reports high accuracy, but inspection of the passing cases reveals that the model is exploiting an artefact of the test data — a phrase, a structural cue — rather than answering the underlying question.[3]

None of these are resolved by switching to a more capable model. All three are architectural: they belong to the framework around the agent, not to the agent itself.

§ 3 — MethodThe four-component framework.

The framework consists of four components, each addressing one of the operational gaps identified in §2. We describe each in turn, with worked examples drawn from the production engagements summarised in Table 1 (§4).

3.1Modular prompt scaffolding.

The prompt is structured as a versioned, modular document with five blocks: task, context, tools, constraints, and examples. Each block can be revised independently. The scaffold is checked into the same repository as the agent code and reviewed under the same code-review process. We treat prompt revisions as code changes — diffs reviewed, commits annotated, regressions caught by the eval harness before deployment.

This is not novel; what we found important is the versioning discipline. A prompt is never edited in production. Every change ships with a version bump and an eval delta. We have rejected prompt edits whose eval delta was within noise — even when the human reviewer felt the new prompt was "clearer" — on the principle that uninspected drift is the leading cause of production regressions.[4]

3.2Typed tool-call contracts.

Each tool exposes a typed contract: input schema, output schema, error taxonomy, and (critically) a negative-example set. The orchestrator validates both successful and failed tool calls against the contract before they reach the agent. A tool call that does not satisfy the negative-example set is treated as a contract violation regardless of the model's reported success.

contract.tsagent-eval / contracts
export const searchInternalDocs: ToolContract = {
  name: "search_internal_docs",
  input: z.object({
    query: z.string().min(1),
    locale: z.enum(["ja", "en"]),
    topK: z.number().int().min(1).max(20)
  }),
  output: z.object({
    docs: z.array(DocSchema),
    confidence: z.number().min(0).max(1)
  }),

  // negative examples — must NOT pass without explicit reason
  negativeExamples: [
    { input: { query: "", locale: "ja", topK: 5 },
      mustReject: "empty_query" },
    { input: { query: "<script>", locale: "ja", topK: 5 },
      mustReject: "injection_pattern" },
    { input: { query: "valid", locale: "ja", topK: 9999 },
      mustReject: "topK_out_of_range" }
  ],

  errorTaxonomy: [
    "empty_query", "injection_pattern",
    "topK_out_of_range", "upstream_timeout",
    "corpus_unavailable"
  ]
};

The negative-example set is the single most useful piece of the contract. It encodes what should fail, which forces the team to reason about failure surfaces before deployment. We collect negative examples from incidents in production — every failure, once root-caused, becomes a negative example for the contract — so the contract grows monotonically with operational experience.

3.3Evaluation harness with failure-mode buckets.

Conventional evaluation reports aggregate accuracy. We instead bucket eval cases by the failure mode they target: drift, injection, multilingual, out-of-corpus, ambiguity, refusal-correctness. Each bucket has its own pass-rate threshold and is gated independently. A regression in any single bucket blocks deployment, even if aggregate accuracy is unchanged.

The bucketing is the difference between a passing eval and a useful one. A 92% aggregate accuracy is meaningless if the 8% failures concentrate in the bucket the deployment cares about. Bucket-level gates make that concentration impossible to overlook.

3.4Operational rituals.

Deployment is not the end of the framework but its operational entry point. Every deployed agent runs four standing rituals:

  • Structured logging of every prompt-input-output triple, indexed by version, with hashed user identifiers for privacy
  • Replay tooling that lets any production trace be re-run against any prompt or agent version, locally or in CI
  • Human-in-loop overrides on every action above a configured cost or risk threshold (configurable per engagement)
  • Weekly incident review in which any failed trace is root-caused and either added to a contract negative-example set or to an eval bucket
The single highest-leverage practice we have adopted is making every production incident produce a deterministic test in the harness within 24 hours. The agent gets harder to break with every break.

§ 4 — EvaluationFour production engagements.

The framework has been deployed across four production engagements. We summarise the engagements anonymised by industry and outcome class, with the operational metrics that drove decisions during the deployment.

All four engagements operational at time of writing. Reported metrics are aggregated and anonymised per client agreement.

Across the four engagements we observed three results worth reporting. First, build-to-deploy time averaged 7.75 weeks across the four engagements, against an internal estimate of 8.0 weeks set before the framework was applied — within noise of estimate. Second, thumbs-up satisfaction on production traffic stabilised between 76% and 82% across the four engagements, with the variation driven primarily by the difficulty of the underlying corpus rather than by the agent itself. Third, operational incidents per agent per month declined from 11 (engagement #03, pre-framework v0.1) to 2 (engagement #01, post-framework v0.4), a 5.5× reduction we attribute primarily to the bucketed eval and the contract negative-example sets.

§ 5 — Failure modesWhat still breaks.

The framework reduces but does not eliminate operational failures. We document the residual failure modes here, both for honesty and to scope future work.

  1. Domain depth. Where the corpus contains genuinely uncommon technical content (rare manufacturing protocols, niche legal language), the agent answers fluently but incorrectly. Bucketing helps surface these but does not solve them. Resolution requires domain-specific fine-tunes or retrieval-enhanced grounding,[5] both outside the framework's scope.
  2. Multi-turn drift. Agents that hold long conversations exhibit slow drift in instruction-following over many turns. Our current mitigation is a periodic re-grounding prompt, applied every n turns. The mitigation is brittle and we expect to revise it in v0.5.
  3. Compliance signal coverage. Compliance constraints (industry-specific, jurisdictional) are not natively bucketed in the harness. We currently encode them as constraint blocks in the prompt scaffold, which works but is unsatisfying. v0.5 will add a dedicated compliance bucket type to the harness.

§ 6 — DiscussionWhat this doesn't do.

The framework is operational, not research-level. It does not advance the state of the art on agent planning, tool-use, or reasoning. It does the unglamorous work of taking a working agent demo and putting it on a deployment surface where it can be observed, evaluated, and revised under production conditions. We expect the framework to be largely uncontroversial to teams that ship language-model systems for paying clients; we publish it because the components are not, in our experience, written down anywhere in one place, and the negative-example discipline is — anecdotally — the part most teams have not converged on.

Limits we want to be explicit about: (a) all four engagements are JP-language enterprise deployments; cross-cultural / cross-language generalisation is not established. (b) The bucket structure has accumulated organically; we do not yet have a principled methodology for bucket design. (c) The harness is open source but the eval datasets used in §4 are not, for client-confidentiality reasons. We release synthetic substitutes alongside the harness for reproducibility.

§ 7 — ConclusionThe framework, in one sentence.

An agent that is shipped, observed, and revised against bucketed evaluation under typed tool contracts will outlive an agent that is shipped against aggregate accuracy. The framework is the engineering substrate that makes that possible without superhuman discipline. The harness is at github.com/xrd-technologies/agent-eval. We welcome contributions, especially from teams operating in non-JP languages.
